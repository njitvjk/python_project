SOLID Principles in Python

Posted on November 23, 2021 by Now-U-Know. Image Credit:Kowsalya Vijayan

Python has become more famous and used in more and more projects and increasingly for more complex application rather than just a scripting language. In this article let us learn how to implement SOLID principles for Python.

The SOLID principles were introduced by Robert C. Martin in his 2000 paper “Design Principles and Design Patterns.These principles help us to write more understandable and maintainable software, as the applications grow in size , we can

reduce their complexity.The image above shows the five concepts of the SOLID principles.

Let us demonstrate the principles using a calculator program

Single Responsibility

"A class should do one thing and therefore it should have only a single reason to change"

  • Testing-A class with one responsibility will have fewer test cases.

  • Lower coupling-Less functionality in a single class will have fewer dependencies.

  • Organization- – Smaller, well-organized classes are easier to search than monolithic ones.

In this code, illustrates the calculator program which can perform operations and store the results in history.However, this code violates the single responsibility principle.

to fix this let us move the all methods related to history to another class

Awesome.Now we have relieved the Calculator class to perform only calculation and let Calculator_history take care of history.

Open for Extension and closed for Modification

"classes should be open for extension but closed for modification. In doing so, we stop ourselves from modifying existing code and causing potential new bugs."

Now, let us explore the concept with the calculator.I used the program for some days and later on, I felt that I need a logger to check the issues.If I add the function into the Calculator Class it changes the logic which is violation.

To fix this, I implement a subclass Calculator which extends the Calculator and extends its functionality to implement logger.

Liskov Substitution Principle(LSP)

"If S is a subtype of T, then objects of type T in a program may be replaced with objects of type S without altering any of the desirable properties of that program."

if class A is a subtype of class B, we should be able to replace B with A without disrupting the behavior of our program.This principle aims to enforce consistency so that the parent Class or its child Class can be used in the same way without any errors.

When a child Class cannot perform the same actions as its parent Class, this can cause bugs.

The child Class should be able to process the same requests and deliver the same result as the parent Class or it could deliver a result that is of the same type.

If the child Class doesn’t meet these requirements, it means the child Class is changed completely and violates this principle.

Interface Segregation

"larger interfaces should be split into smaller ones. By doing so, we can ensure that implementing classes only need to be concerned about the methods that are of interest to them.in other words, A client should not depend on methods it does not use"

The calculator program can be used by different roles like students and general people. Students might also do trignometric functions where as general people will only need addition /subtraction/division or multiplication.Our program has all functionalities together and some unwanted functions for different actors. This is violation of the principle. Not let us split the program into two one for student and other for general people

The school calculator extends basic methods addition, division,subtraction,multiplication and have trigonometric methods extended from inbuilt Python Class

Dependency Inversion

"The principle of dependency inversion refers to the decoupling of software modules. This way, instead of high-level modules depending on low-level modules, Both will depend on abstractions."

Now we have two Class CommonCalculator and a SchoolCalculatorBoth classes use abstraction for adding the result to history as per rule,in this way both will not be dependent on each other

Conclusion

So far, we have discussed these five principles and highlighted their goals. They are to help you make your code easy to adjust, extend and test with little to no problems.
